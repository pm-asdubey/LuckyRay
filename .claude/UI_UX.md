# UI_UX.md

# LuckyRay UI / UX Specification

## Purpose

LuckyRay should feel like a premium modern software product, not a traditional astrology website.

The interface should inspire confidence, curiosity and calm.

Users should immediately feel that they are using a thoughtfully designed product rather than a collection of screens.

Every interaction should reinforce quality.

---

# Overall Experience

LuckyRay should feel:

* Elegant
* Intelligent
* Modern
* Calm
* Premium
* Minimal
* Fast
* Trustworthy

The interface should subtly communicate the mystery of Jyotish without becoming theatrical.

Avoid visual clichés commonly associated with astrology.

---

# Emotional Goals

Users should feel:

"I want to explore."

"I trust this."

"This feels premium."

"This feels intelligent."

"I understand what I'm seeing."

Never:

"This feels fake."

"This feels cluttered."

"This feels overwhelming."

---

# Design Inspiration

Draw inspiration from the quality and polish of:

* Linear
* Raycast
* Notion
* Arc Browser
* Apple Human Interface Guidelines

Do **not** copy layouts or branding.

Instead emulate:

* consistency
* spacing
* responsiveness
* attention to detail
* simplicity

---

# Visual Identity

The visual language should balance:

Technology

*

Ancient knowledge

Use modern typography combined with subtle astronomical themes.

The application should not resemble a religious website.

---

# Color Philosophy

Dark mode is the primary experience.

Light mode may be added later.

Use restrained colors.

Accent colors should be used sparingly.

Avoid rainbow palettes.

Avoid highly saturated gradients.

Gold, indigo, deep blue and muted neutrals may be explored carefully, but the interface should remain understated.

---

# Typography

Typography should communicate trust.

Prefer modern, highly readable fonts.

Use a clear hierarchy:

Display

Heading

Subheading

Body

Caption

Avoid decorative fonts.

---

# Layout

Use generous whitespace.

Avoid dense information.

Every screen should have a clear visual hierarchy.

Users should know where to look first.

---

# Navigation

Navigation should remain simple.

Maximum primary sections:

* Home
* Profiles
* Chat
* Charts
* Settings

Avoid deeply nested navigation.

Users should reach any major function within two interactions.

---

# Mobile Experience

The application must be fully usable on mobile browsers.

Do not remove functionality.

Instead:

Stack layouts.

Collapse secondary panels.

Use drawers where appropriate.

Ensure tap targets remain comfortable.

Charts and conversations should remain readable on small screens.

---

# Home Screen

Purpose:

Welcome the user.

Allow profile selection.

Provide quick access to recent conversations.

Display recent charts.

Avoid information overload.

---

# Profile Management

Users should easily:

Create profile.

Edit profile.

Delete profile.

Duplicate profile.

Switch between profiles.

The interaction should feel lightweight.

---

# Birth Details Screen

Collect:

Date

Time

Location

Provide validation.

Provide timezone handling.

Provide helpful guidance for uncertain birth times where applicable.

---

# Chart Screen

The chart screen should become the user's primary workspace.

Include:

Birth chart visualization.

Planet summary.

House summary.

Current dasha.

Key yogas.

Important observations.

Allow users to expand details progressively.

Avoid overwhelming first-time users.

---

# AI Chat

The chat should feel like a premium messaging interface.

Requirements:

Streaming responses.

Markdown support where appropriate.

Code blocks unnecessary.

Tables where beneficial.

Conversation history.

Auto-scroll.

Message timestamps optional.

Allow users to revisit previous discussions.

---

# Progressive Disclosure

Begin with simple summaries.

Allow users to expand into deeper technical explanations.

Never present every calculation immediately.

Users should discover complexity naturally.

---

# Animations

Animations should communicate state.

Examples:

Loading.

Expansion.

Navigation.

Chart transitions.

Avoid decorative motion.

Keep animations subtle and fast.

Respect reduced-motion preferences.

---

# Loading States

Every asynchronous action should provide feedback.

Examples:

Skeletons.

Progress indicators.

Streaming text.

Never leave users wondering whether the application has frozen.

---

# Empty States

Empty screens should educate.

Examples:

Explain what the feature does.

Guide the user toward the next action.

Avoid generic placeholders.

---

# Error States

Errors should:

Explain what happened.

Explain what the user can do next.

Avoid technical jargon.

Never expose stack traces.

---

# Accessibility

Support:

Keyboard navigation.

Readable typography.

High contrast.

Focus indicators.

Screen readers where practical.

Large tap targets.

Accessibility is part of quality.

---

# Performance Perception

The application should feel fast even when work is occurring.

Use:

Streaming.

Skeletons.

Optimistic transitions where appropriate.

Lazy loading for non-critical content.

---

# Visual Components

Components should appear consistent.

Cards.

Buttons.

Dialogs.

Inputs.

Dropdowns.

Tabs.

Accordions.

Badges.

Charts.

Every component should share the same spacing, radius, typography and interaction philosophy.

---

# Responsiveness

Support:

Desktop.

Laptop.

Tablet.

Mobile browser.

No horizontal scrolling.

No broken layouts.

No unusable controls.

---

# User Journey

The ideal experience:

1. Open LuckyRay.
2. Create a profile in under two minutes.
3. Generate a chart.
4. View key observations immediately.
5. Ask a natural-language question.
6. Receive an intelligent, evidence-based answer.
7. Continue exploring naturally.

The product should encourage curiosity without overwhelming the user.

---

# Future Design Considerations

Design the UI so future additions integrate naturally:

Voice interaction.

Compatibility analysis.

Transit dashboards.

Notifications.

Desktop packaging.

Offline AI.

Multi-language.

Do not hard-code layouts that prevent future expansion.

---

# Definition of Success

A first-time user should describe LuckyRay as:

* Beautiful.
* Easy to use.
* Fast.
* Premium.
* Trustworthy.
* Thoughtfully designed.

The interface should quietly disappear behind the experience, allowing users to focus on understanding their chart rather than learning how to use the application.
